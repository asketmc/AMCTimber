# Verify AMCTimber Releases

AMCTimber releases are published as GitHub Release assets. The release workflow builds the jar from the
tagged source, generates checksums and SBOMs, signs artifacts with Sigstore/cosign keyless signing, and
creates GitHub artifact attestations.

This is verification evidence, not a formal security certification.

## Expected Assets

For a release tag such as `v1.0.4`, expect:

- `AMCTimber-1.0.4.jar`
- `SHA256SUMS.txt`
- `sbom.spdx.json`
- `sbom.cdx.json`
- `jar-safety-report.txt`
- `*.sigstore.json` bundles for each release asset
- GitHub artifact attestations for the release assets

## Check The Jar Hash

Linux/macOS:

```bash
sha256sum -c SHA256SUMS.txt --ignore-missing
```

Windows PowerShell:

```powershell
$jar = "AMCTimber-1.0.4.jar"
$expected = (Select-String -Path SHA256SUMS.txt -Pattern $jar).Line.Split(" ")[0].ToLowerInvariant()
$actual = (Get-FileHash -Algorithm SHA256 $jar).Hash.ToLowerInvariant()
if ($actual -ne $expected) { throw "checksum mismatch" }
```

## Verify The Sigstore Bundle

Install `cosign`, then verify a downloaded jar and its bundle:

```bash
cosign verify-blob \
  --bundle AMCTimber-1.0.4.jar.sigstore.json \
  --certificate-identity-regexp 'https://github.com/asketmc/AMCTimber/.github/workflows/release.yml@refs/tags/v.*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  AMCTimber-1.0.4.jar
```

Repeat for `SHA256SUMS.txt`, SBOMs, and `jar-safety-report.txt` if you need full release-asset coverage.

## Verify GitHub Artifact Attestation

Install GitHub CLI and run:

```bash
gh attestation verify AMCTimber-1.0.4.jar \
  --repo asketmc/AMCTimber \
  --source-ref refs/tags/v1.0.4
```

The attestation should bind the jar to this repository, the release workflow, and the release tag.

## Inspect SBOMs

The release includes two SBOM formats:

- `sbom.spdx.json` for SPDX-compatible tooling.
- `sbom.cdx.json` for CycloneDX-compatible tooling.

At minimum, confirm both files are present, covered by `SHA256SUMS.txt`, and signed with corresponding
`*.sigstore.json` bundles.

## Inspect The Jar Safety Report

`jar-safety-report.txt` is generated by `scripts/check-release-jar.sh`. It fails the release if the jar
contains:

- native binaries: `.dll`, `.so`, `.dylib`, `.exe`
- script files: `.bat`, `.cmd`, `.ps1`, `.vbs`, `.sh`
- nested `.jar` files
- shaded dependency signature metadata under `META-INF`

This is an artifact hygiene control. It does not replace antivirus scanning, sandbox analysis, or a
manual code review.
